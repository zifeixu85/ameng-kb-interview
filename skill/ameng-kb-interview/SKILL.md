# 采访式个人知识库

把第一次建库从“复制一堆空文件夹”改成“先了解这个人，再长出适合他的结构”。

## 开始前读取

始终读取：

- `references/vault-architecture.md`：固定骨架与可选模块。
- `references/routing-and-privacy.md`：写入位置、隐私与确认边界。
- `references/note-contract.md`：笔记属性、来源和双链契约。

开始提问前读取 `references/interview-playbook.md`。初始化新库、接入旧库或恢复采访时读取
`references/first-run-and-resume.md`。运行脚本前读取 `references/structure-plan-schema.md`。

## 核心原则

1. 先采访，再决定可选目录；不要把产品、客户、内容、黑客松等目录默认塞给所有人。
2. 固定骨架只保留 `00 / 10 / 20 / 30 / 40 / 99` 六个顶层职责；`20` 与 `30` 下的具体模块由采访决定，不创建历史采矿场、旧库或归档冷库。
3. 一次只问一个问题。第一个正式采访问题必须先了解用户当下在做什么，不要要求用户一开始就定义“知识库用途”。
4. 用户的回答比预设问卷重要。询问用途、输入、输出或动态模块时，根据前文给出 2–3 个候选，并保留“都不是”和“请你推荐”出口；不让用户凭空设计系统。
5. 根据答案追问时间、场景、行动、证据和原话。先保存原始采访，再派生结构化笔记；原话和整理稿不能混成一个“AI 总结”。
6. 私密自我与对外表达分开。任何可能公开使用的笔记都不得自动带入 private 细节。
7. 新派生资产默认 `status: 待确认`。用户确认后才能标记为 `可复用`。
8. 不覆盖已有文件，不删除旧目录，不批量移动用户资料。已有 Vault 只做增量接入。
9. 知识库负责上下文、判断、复盘和资产；任务软件与日历只是可选执行层，不是建库前置条件。

## 执行流程

### 1. 确认工作区

确认目标 Vault 的绝对路径，并判断：

- 新建空 Vault；
- 已有 Vault，但尚无清晰结构；
- 已有结构，只想补做采访和基础资料。

若当前目录不明确，只问这一项，不要猜路径。

### 2. 选择采访深度

提供两个选项：

- **快速版（约 15–20 分钟）**：确定结构，生成 6–10 篇基础笔记，跑通第一次写入。
- **完整版（约 45–60 分钟）**：补充项目、学习、内容、产品/客户等条件分支，形成更完整的首版知识库。

快速版不是“让用户填一份精简问卷”。先问他当下在做什么、哪类事情经常重复，再由 AI 给出最多 3 个首个闭环候选。

### 3. 已有 Vault 的只读接入门

如果是已有 Vault，在创建采访记录或任何文件之前：

1. 只读列出顶层目录、`AGENTS.md` 和必要入口，不扫描私人正文；
2. 把现有目录映射到六个固定职责，列出计划新增的文件；
3. 明确说明不会移动、改名、覆盖或删除什么；
4. 等用户确认这份映射与写入边界。

在这次确认之前保持完全只读。新建空 Vault 可以跳过本步。
确认后，本 Skill 和 references 中出现的 `00 / 10 / 20 / 30 / 40 / 99` 都是逻辑标准路径；已有 Vault 每次读写前必须先经过 `existing_mappings` 换算，不得另行创建重复的标准目录。

### 4. 建立原始采访记录

创建或继续：

```text
00-系统工作台/采访记录/YYYY-MM-DD-首次建库采访.md
```

上面是新 Vault 的标准路径。已有 Vault 必须使用第 3 步确认的映射路径；例如 `00-系统工作台 → Admin/System` 时，实际写入 `Admin/System/采访记录/...`。

逐题追加问题与用户原话，并记录当前阶段。暂停时保留：已完成阶段、下一题、已生成文件、待确认项。

### 5. 逐题采访

按 `references/interview-playbook.md` 执行。顺序围绕：

1. 他当下在做什么，主要身份和精力放在哪里；
2. 他已经重复做过哪些事，最近哪个摩擦值得先解决；
3. AI 基于前两步提出首个使用闭环候选，与用户确认一个真实输入和具体产物；
4. 他有哪些重要经历、能力、真实项目和承诺；
5. 平时如何学习、记录和复盘，有哪些可复用故事、判断与方法；
6. 是否存在内容、产品、客户、合作、主题分享等可选场景；
7. 哪些内容绝不能被公开调用。

用户没有相关场景时跳过整条分支，不为凑完整而继续追问。

### 6. 生成结构方案并确认一次

先展示两部分：

- 固定骨架：所有人都会创建的目录与入口文件；
- 动态模块：根据本次回答建议创建的目录，并逐项说明“为什么需要”。

同时生成符合 `references/structure-plan-schema.md` 的 JSON 方案。只在用户确认结构方案后批量创建目录和笔记。对已有 Vault，这一步确认采访后长出的动态模块；第 3 步的写入路径必须早已确认。

### 7. 初始化或增量接入

优先运行：

```bash
python3 scripts/init_vault.py "/absolute/path/to/vault" --plan "/path/to/structure-plan.json"
```

先用 `--dry-run` 预览。脚本只能创建缺失内容，不能覆盖已有文件。若执行环境没有 Python，则按
`references/vault-architecture.md` 手动创建同等结构。

### 8. 写入首批笔记

按 `references/routing-and-privacy.md` 分流回答，按 `references/note-contract.md` 写属性和双链。

最低交付：

- 1 篇原始采访记录；
- 1 篇建库配置与结构说明；
- 3–5 篇 private 自我上下文；
- 用户确实需要公开表达时，生成 1 篇对外角色与表达草稿；
- 2–5 篇有采访证据的原子化素材、方法、学习或项目笔记；
- 有真实项目时，至少 1 篇项目母舰；
- 有可选场景时，生成对应模块的首批笔记，而不是只建空目录。

每篇派生笔记链接回采访原文；采访原文增加“派生笔记”清单，形成真实双链。

### 9. 跑通第一次使用

不要停在“目录已经建好”。选择用户最高频的一项任务，完成一次真实输出，例如：

- 根据个人资料填写一份活动报名表；
- 从项目与方法笔记生成一篇内容初稿；
- 从项目母舰提取下一步行动候选；
- 从产品、客户和流程资料生成一版 FAQ 草稿。

把输出结果和反馈写回对应来源笔记。外部发布、客户消息、日历事件和任务批量创建仍需单独授权。

### 10. 验证与汇报

运行：

```bash
python3 scripts/validate_vault.py "/absolute/path/to/vault"
```

最后汇报：

- 创建与跳过的目录；
- 生成的笔记及路径；
- 启用的动态模块；
- private / internal / public 分流结果；
- 待确认项；
- 建议用户本周继续喂入的第一个真实输入。

## 禁止事项

- 不因 A梦本人参加过黑客松，就为其他用户创建黑客松目录。
- 不把原始采访直接润色成“用户已经认可的方法论”。
- 不把日记、健康、家庭、关系或财务压力放进可公开调用的资产库。
- 不把来源当成文件夹分类；按主题和用途组织，来源写进属性。
- 不把任务堆进 Obsidian 正文。只记录项目上下文和下一步候选。
- 不宣称已经完成自动化、发布或商业结果；只汇报实际生成与验证的内容。
