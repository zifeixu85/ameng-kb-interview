# 首次使用、已有 Vault 与暂停续访

## 新建 Vault

1. 确认目标目录允许写入。
2. 先采访到能确定结构，不先复制完整模板。
3. 生成结构方案并获得一次确认。
4. 用 `init_vault.py --dry-run` 预览，再正式创建。
5. 打开 Obsidian 后，把附件目录设置为 `99-附件库/_inbox`。
6. 完成一次真实输入到输出，验证这套结构不是空壳。

## 已有 Vault

本段在创建采访记录或任何其他文件之前完成；用户确认映射前必须保持只读。

1. 读取顶层目录、已有规则、MOC / README / 索引和用户指定的排除路径。
2. 按建库目标渐进式读取相关 Markdown 正文，包括必要的 `private` 笔记；不默认遍历无关原始库和大型附件。
3. 输出“已知信息 / 待确认 / 冲突 / 缺失”清单，每条保留来源。
4. 找出与六个固定职责等价的现有目录，在结构方案的 `existing_mappings` 中记录复用路径。
5. 列出会直接复用的现有笔记、只需确认的笔记和确实需要新建的文件。
6. 不自动迁移、改名、合并、覆盖或删除。
7. 初始化脚本遇到同名文件一律跳过，并在报告中列出。

映射确认后，只对待确认、冲突、缺失或过期信息做补充采访。后续采访记录、自我上下文、资产、项目和学习输入的每一次读写都先应用 `existing_mappings`。文档中的标准路径只是逻辑名称，不能在已有 Vault 中另行创建重复目录。

## 暂停与恢复

原始采访记录 frontmatter 保存 `interview_stage`。正文末尾维护：

```markdown
## 采访进度

- 已完成：阶段 0、阶段 1
- 下一题：阶段 2 / 当前有哪些持续一周以上的事情？
- 已生成：[[...]]
- 待确认：内容模块是否启用
```

用户说“暂停”时：先写入进度，再结束当前回合。用户说“继续采访”时：新 Vault 先读取最新采访记录；已有 Vault 同时读取已确认的来源索引。只复述最后一个已确认结论和下一项差额，不从头重复。

## 冲突处理

- 目标文件不存在：创建。
- 目标文件存在但为空：先询问是否补充。
- 目标文件存在且有内容：新建 `标题-采访补充-YYYY-MM-DD.md`，不覆盖。
- 目录职责冲突：记录在结构方案中，暂停批量写入，只处理不冲突部分。
- 多篇笔记对同一事实说法不同：并列来源并询问用户以哪一版为当前事实，不由 AI 静默合并。
- 旧笔记描述当前角色、项目状态、优先级或对外计划：标为“疑似过期”，用确认题核对；不能只因日期旧就否定稳定经历。
- 用户本次回答修正旧笔记：新记录写明“当前以本次确认优先”，保留旧笔记作为历史，不自动覆盖原文。

## 结束标准

新 Vault 只有同时满足以下条件才算首次建库完成；已有 Vault 可用合格的现有来源替代“原始采访”和“首批新笔记”，但仍要完成来源索引、分流、双链与一次真实输出：

- 有原始采访或现有笔记来源；
- 有结构方案；
- 有首批结构化笔记，而非只有目录；
- 有 private / internal 分流；
- 有来源双链；
- 跑通过一次真实输出；
- 用户知道下一份输入放哪里、如何继续更新。
